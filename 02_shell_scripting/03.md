# Module 3 — Variables

## What You Will Learn
- What a variable is

- How to reference variables with `$`

## Why This Matters

Right now `system-info.sh` has the path `/` hardcoded. If you want to check a different disk, you have to edit the script. If you use a variable, you change one line at the top — the rest of the script updates automatically. Variables are the foundation of flexible, reusable scripts.

## Real Linux Problem

InfraOps runs the same backup script on three environments: `dev`, `staging`, and `prod`. Without variables, you need three separate scripts. With variables, you have one script and you just change the environment name at the top.

## Simple Explanation

A variable is a named storage box. You put a value in the box, give the box a label, and then use the label whenever you need the value.

```
Variable name        Value stored inside
     │                      │
     ▼                      ▼
ENVIRONMENT    =     "production"

Later in the script:
echo "Running on: $ENVIRONMENT"
          ▲
          │
          $ means "look up the variable"
```

## Creating Variables

```bash
# Basic variable assignment
# IMPORTANT: No spaces around the = sign
NAME="Alice"
SERVER="web01"
ENVIRONMENT="dev"
PORT=8080

# WRONG - these all fail:
NAME = "Alice"   # spaces around =
name="Alice"     # lowercase works but breaks convention
```

**Naming conventions for scripts:**

- Use `UPPERCASE` for variables you set yourself (makes them easy to spot)
- Use letters, numbers, and underscores only
- Start with a letter (never a number)
- Descriptive names: `BACKUP_DIR` not `bd`

## Using Variables

```bash
#!/bin/bash

SERVER="web01"
ENVIRONMENT="production"
ADMIN="ops@infraops.com"

echo "Server name : $SERVER"
echo "Environment : $ENVIRONMENT"
echo "Contact     : $ADMIN"

# Variables inside longer strings - use curly braces for clarity
echo "Checking ${SERVER}-backup.tar.gz"
```

The `${}` syntax (curly braces) is used when a variable name is next to other text without a space. It tells Bash exactly where the variable name ends.

```bash
SERVER="web01"

echo "$SERVERbackup"    # WRONG: Bash looks for a variable called SERVERbackup
echo "${SERVER}backup"  # CORRECT: outputs web01backup
```

## Guided Example — welcome-server.sh

```bash
#!/bin/bash

# welcome-server.sh
# Displays a personalized server welcome banner
# Uses variables for easy customization

COMPANY="InfraOps Ltd."
ENVIRONMENT="Development"
ADMIN_EMAIL="ops@infraops.com"
MAX_DISK_WARN=80

echo "=========================================="
echo "  $COMPANY"
echo "  Environment: $ENVIRONMENT"
echo "=========================================="
echo "Admin contact : $ADMIN_EMAIL"
echo "Disk warning  : ${MAX_DISK_WARN}% usage"
echo "Checked at    : $(date)"
echo "=========================================="
```

Notice the last `echo` uses `$(date)` — this is called **command substitution**. It runs a command and captures its output as a value (we cover this fully in Module 5). Variables and command substitution work exactly the same way once you understand the pattern.

## Common Mistakes

**Mistake 1: Spaces around `=`**
```bash
NAME = "Alice"   # Error: NAME: command not found
NAME="Alice"     # Correct
```

**Mistake 2: Forgetting the `$`**
```bash
NAME="Alice"
echo "Hello NAME"    # Prints: Hello NAME
echo "Hello $NAME"   # Prints: Hello Alice
```

**Mistake 3: Quoting with single quotes**
```bash
NAME="Alice"
echo 'Hello $NAME'   # Prints: Hello $NAME (variables don't expand in single quotes)
echo "Hello $NAME"   # Prints: Hello Alice (double quotes expand variables)
```

Single quotes: literal (no variable expansion). Double quotes: variables expand.

## Quick Recap

- Variables store values with `NAME="value"` (no spaces around `=`)
- Access them with `$NAME` or `${NAME}`
- Double quotes expand variables; single quotes do not
- Command substitution `$(command)` captures output into a variable

## Interview Corner — Variables

**Why interviewers ask:** Variables show whether you write maintainable scripts or hardcoded ones. Interviewers give you a script with hardcoded paths and ask "how would you improve this?"

**What they expect:**
- You can create and reference variables correctly
- You understand the difference between `$VAR`, `${VAR}`, and `'$VAR'`
- You know when to use `readonly` for constants

**Common beginner mistakes:**
- Spaces around `=` (breaks variable assignment)
- Forgetting `$` when using variables
- Confusing single and double quotes

**Practice question:** "What's the output of this script?"
```bash
X="hello"
Y="world"
echo '$X $Y'
echo "$X $Y"
echo "${X}_${Y}"
```

Answer: `$X $Y` then `hello world` then `hello_world`
