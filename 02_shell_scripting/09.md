## Why This Matters

Every script you've written so far runs from top to bottom once. As scripts grow, the same blocks of code repeat. Change one, and you have to remember to change all the others. Functions solve this by letting you write code once and reuse it everywhere — even across multiple scripts.

## Real Linux Problem

Every script in `devops-bash-toolkit` prints a banner. Right now each script has its own copy of the banner code. If the company name changes, someone has to update every single script. With a function in a shared library file, you change it once.

## Simple Explanation

A function is a named block of commands. You define it once, then call it by name whenever you need it.

```
Define (write once):          Call (use many times):
────────────────────          ──────────────────────
print_banner() {              print_banner    ← runs the block
    echo "InfraOps"           print_banner    ← runs it again
    echo "$(date)"            print_banner    ← and again
}
```

## Function Syntax

```bash
# Define a function
function_name() {
    # commands
}

# Call it
function_name

# With arguments
greet() {
    echo "Hello, $1!"
    echo "Your environment is: $2"
}

greet "Alice" "production"
# $1 = "Alice", $2 = "production"
```

Inside a function, `$1`, `$2`, `$3` refer to the arguments passed to that function — not to the script's original arguments.

## Returning Values

Bash functions don't "return" data the way most languages do. Instead there are two patterns:

```bash
# Pattern 1: Print the value and capture it
get_disk_usage() {
    df / | awk 'NR==2 {print $5}' | tr -d '%'
}

USAGE=$(get_disk_usage)   # Capture the printed output
echo "Disk: $USAGE%"

# Pattern 2: Use a global variable
get_hostname() {
    RESULT=$(hostname)   # Write to a global-ish variable
}

get_hostname
echo "Host: $RESULT"
```

## Guided Example — lib/utils.sh

Create a shared utilities library:

```bash
#!/bin/bash

# lib/utils.sh
# Shared utility functions for the InfraOps toolkit
# Source this file in other scripts with: source lib/utils.sh

# Print a formatted section header
print_header() {
    local TITLE="$1"
    local WIDTH=44
    echo ""
    printf '%.0s=' $(seq 1 $WIDTH)
    echo ""
    echo "  $TITLE"
    printf '%.0s=' $(seq 1 $WIDTH)
    echo ""
}

# Print a status line: OK, WARN, or FAIL
print_status() {
    local STATUS="$1"
    local MESSAGE="$2"
    printf "%-6s %s\n" "[$STATUS]" "$MESSAGE"
}

# Check if a command is available on this system
command_exists() {
    local CMD="$1"
    if command -v "$CMD" &>/dev/null; then
        return 0   # success = true
    else
        return 1   # failure = false
    fi
}

# Get current timestamp in a consistent format
get_timestamp() {
    date "+%Y-%m-%d %H:%M:%S"
}
```

**Using the library in another script:**

```bash
#!/bin/bash

# Source the library (like importing it)
source ~/devops-bash-toolkit/lib/utils.sh

print_header "System Information"
print_status "OK" "System is running"
print_status "WARN" "High memory usage detected"

TIMESTAMP=$(get_timestamp)
echo "Report time: $TIMESTAMP"
```

## Guided Example — Refactoring disk-check.sh

Now use functions to clean up `disk-check.sh`:

```bash
#!/bin/bash

source ~/devops-bash-toolkit/lib/utils.sh

# Get disk usage percentage as a number
get_disk_percent() {
    df "$1" | awk 'NR==2 {print $5}' | tr -d '%'
}

# Check one mount point
check_mount() {
    local MOUNT="$1"
    local WARN=70
    local ALERT=90

    local USAGE
    USAGE=$(get_disk_percent "$MOUNT")

    if [ "$USAGE" -ge "$ALERT" ]; then
        print_status "FAIL" "$MOUNT — ${USAGE}% (CRITICAL)"
    elif [ "$USAGE" -ge "$WARN" ]; then
        print_status "WARN" "$MOUNT — ${USAGE}% (elevated)"
    else
        print_status "OK" "$MOUNT — ${USAGE}%"
    fi
}

print_header "Disk Usage Report — $(get_timestamp)"

for MOUNT in / /var /tmp; do
    if [ -d "$MOUNT" ]; then
        check_mount "$MOUNT"
    fi
done
```

Notice `local` before variable declarations inside functions. Using `local` keeps those variables from accidentally overwriting same-named variables elsewhere in the script. Always use `local` inside functions.

## Common Mistakes

**Calling a function before defining it:**
```bash
greet         # Error: greet: command not found
greet() {
    echo "hello"
}
```
Always define functions before calling them. Or put all function definitions at the top.

**Forgetting `local`:**
```bash
check_disk() {
    USAGE=50    # No local — overwrites any global USAGE variable!
}
```
```bash
check_disk() {
    local USAGE=50   # Safe — stays inside the function
}
```

**Trying to return a string with `return`:**
```bash
get_hostname() {
    return $(hostname)   # WRONG — return only handles numbers 0-255
}
# Correct way:
get_hostname() {
    echo $(hostname)     # Print it
}
HOSTNAME=$(get_hostname)  # Capture it
```

## Quick Recap

- Define: `function_name() { ... }`
- Call: `function_name arg1 arg2`
- Inside the function: `$1`, `$2` = arguments
- Use `local` for all variables inside functions
- Return data by printing it and capturing with `$()`
- Source shared libraries with `source file.sh`

## Interview Corner — Functions

**Why interviewers ask:** Functions test whether you write maintainable code or copy-paste code. Senior engineers immediately ask "is this logic reusable?" Interviewers want to see you think the same way.

**What they expect:**
- You can define, call, and pass arguments to functions
- You know why `local` matters
- You can explain the "print and capture" pattern for returning values
- You've used `source` to share functions across scripts

**Practice question:** "Write a function that checks if a service is running and returns a status string."

```bash
check_service() {
    local SERVICE="$1"
    if systemctl is-active --quiet "$SERVICE" 2>/dev/null; then
        echo "RUNNING"
    else
        echo "STOPPED"
    fi
}

STATUS=$(check_service nginx)
echo "nginx is: $STATUS"
```
