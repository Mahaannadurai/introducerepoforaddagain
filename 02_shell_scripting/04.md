# Module 4 — Getting User Input

## What You Will Learn
- How to ask a user for input using `read`
- How to prompt the user before reading
- How to use what they typed later in the script
- How to set default values when input is blank
- How to read passwords silently

## Why This Matters

So far your scripts only work with hardcoded values. A truly useful script can be run by anyone — it asks questions and adapts to the answers. InfraOps wants a script that any team member can run without editing code.

## Real Linux Problem

You've written a backup script that archives a specific folder. But every team member backs up a different folder. Instead of everyone editing the script, you want the script to ask: "Which folder should I back up?"

## Simple Explanation

Think of `read` like asking someone to fill in a form. Your script pauses, shows a question, waits for them to type something, then stores what they typed in a variable.

```
Script runs
    │
    ▼
echo "What is your name?"   ← shows the question
read NAME                    ← waits, stores answer in NAME
    │
    ▼
echo "Hello, $NAME"          ← uses the answer
```

## Basic read Syntax

```bash
# Simple read
read NAME

# Read with a prompt (show question on same line as cursor)
read -p "Enter your name: " NAME

# Read without showing typed characters (for passwords)
read -s -p "Enter password: " PASSWORD
echo ""  # newline after silent input

# Read with a default value (if user presses Enter without typing)
read -p "Enter environment [dev]: " ENV
ENV=${ENV:-dev}   # If ENV is empty, use "dev"
```

The `${VARIABLE:-default}` pattern is important. It means: "use the variable's value, but if it's empty or unset, use this default instead."

## Guided Example — welcome-user.sh

```bash
#!/bin/bash

# welcome-user.sh
# Interactive welcome script for InfraOps team members
# Asks for name and preferred environment, then shows a summary

echo "=========================================="
echo "  InfraOps Server Access"
echo "=========================================="
echo ""

read -p "Enter your name: " USER_NAME
read -p "Enter your team [ops]: " USER_TEAM
USER_TEAM=${USER_TEAM:-ops}

read -p "Which environment? (dev/staging/prod) [dev]: " ENVIRONMENT
ENVIRONMENT=${ENVIRONMENT:-dev}

echo ""
echo "=========================================="
echo "  Access Summary"
echo "=========================================="
echo "Name        : $USER_NAME"
echo "Team        : $USER_TEAM"
echo "Environment : $ENVIRONMENT"
echo "Login time  : $(date)"
echo "=========================================="
echo ""
echo "Welcome, $USER_NAME! Have a productive session."
```

**Run it:**
```bash
chmod +x ~/devops-bash-toolkit/scripts/hello/welcome-user.sh
./welcome-user.sh
```

**What happens:**
1. Banner prints
2. Script pauses, shows "Enter your name: " — you type your name
3. Script pauses again — you type your team (or press Enter for "ops")
4. Script pauses — you type an environment (or Enter for "dev")
5. Summary prints with everything you entered

## Common Mistakes

**Forgetting `-p` for inline prompts:**
```bash
echo "Enter name:"
read NAME        # Prompt on one line, cursor on next — looks messy
read -p "Enter name: " NAME  # Clean — prompt and cursor on same line
```

**Not quoting the variable when using it:**
```bash
read -p "Enter path: " DIR_PATH
ls $DIR_PATH     # Breaks if path has spaces
ls "$DIR_PATH"   # Correct — always quote variables used as paths
```

**Reading a password and echoing it:**
```bash
read -s -p "Password: " PASS
echo "You entered: $PASS"  # Never do this — defeats the point of -s
```

## Quick Recap

- `read NAME` stores input in `NAME`
- `read -p "Question: " NAME` shows a prompt on the same line
- `read -s` hides input (for passwords)
- `${VAR:-default}` returns `default` if `VAR` is empty
