# Module 2 — Your First Bash Scripts

## What You Will Learn
- How to create a script file
- What the shebang line is and why it matters
- How to make a script executable with `chmod +x`
- How to run a script
- The difference between running with `bash script.sh` vs `./script.sh`

## Why This Matters

Right now, if you want to print your server's name, IP address, and disk usage, you have to type three separate commands. A script packages all of that into one step. Once you understand how scripts work, everything else in this course clicks into place.

## Real Linux Problem

InfraOps wants a quick way to check what machine they're logged into. No one can remember which server is which. Every time someone logs in, they want a greeting that shows the hostname and current user.

## Simple Explanation

A script file is like a recipe. The recipe tells you:
1. What type of kitchen tools you need (shebang)
2. All the steps to follow (your commands)

The **shebang** (`#!/bin/bash`) is the first line of every Bash script. It tells Linux: "run this file using the Bash program." Without it, Linux might try to run your script using the wrong tool.

```
#!/bin/bash       ← shebang (this line tells Linux: use Bash)
echo "Hello"      ← first command
echo "World"      ← second command
```

**Making a script executable:** Files on Linux have permissions. By default, new files can't be executed. `chmod +x script.sh` adds the execute permission — like giving your recipe card the authority to be cooked.

## Guided Example — hello-infraops.sh

**Step 1:** Create the file

```bash
cd ~/devops-bash-toolkit/scripts/hello
```

Open VS Code:
```bash
code hello-infraops.sh
```

**Step 2:** Type this exactly

```bash
#!/bin/bash

# hello-infraops.sh
# Greets the user when they log in
# Author: Your Name
# Date: 2026-06-09

echo "============================================"
echo "  Welcome to InfraOps Ltd."
echo "============================================"
echo "Hostname : $(hostname)"
echo "User     : $(whoami)"
echo "Date     : $(date)"
echo "============================================"
```

**Every line explained:**

| Line | What It Does |
|---|---|
| `#!/bin/bash` | Tells Linux: run this with Bash |
| `# hello-infraops.sh` | A comment — Linux ignores this |
| `echo "==..."` | Prints a line of equals signs |
| `$(hostname)` | Runs the `hostname` command and inserts its output |
| `$(whoami)` | Runs `whoami` and inserts the result |
| `$(date)` | Runs `date` and inserts the result |

**Step 3:** Save the file, then go back to terminal

**Step 4:** Make it executable

```bash
chmod +x hello-infraops.sh
```

**Step 5:** Run it

```bash
./hello-infraops.sh
```

**Expected output:**
```
============================================
  Welcome to InfraOps Ltd.
============================================
Hostname : myserver
User     : yourname
Date     : Tue Jun  9 08:42:11 UTC 2026
============================================
```

## Why `./` at the start?

When you type `ls`, Linux knows where to find it because `/usr/bin/ls` is in your `PATH` — a list of folders Linux searches for programs. Your home folder is not in PATH, so Linux won't find your script automatically. The `./` means "look in this current folder."

Three ways to run a script:

```bash
# Method 1: Specify the path explicitly
./hello-infraops.sh

# Method 2: Run it with bash directly (no chmod needed)
bash hello-infraops.sh

# Method 3: Use the full path
/home/yourname/devops-bash-toolkit/scripts/hello/hello-infraops.sh
```

## Guided Example — system-info.sh

Create a second script:

```bash
code ~/devops-bash-toolkit/scripts/system/system-info.sh
```

```bash
#!/bin/bash

# system-info.sh
# Displays basic system information
# Used by: InfraOps daily check routine

echo "===== System Information ====="
echo ""
echo "Operating System:"
cat /etc/os-release | grep PRETTY_NAME

echo ""
echo "Kernel Version:"
uname -r

echo ""
echo "Uptime:"
uptime -p

echo ""
echo "Logged In Users:"
who | wc -l

echo ""
echo "Disk Usage (/):"
df -h / | tail -1
```

```bash
chmod +x ~/devops-bash-toolkit/scripts/system/system-info.sh
./~/devops-bash-toolkit/scripts/system/system-info.sh
```

## Common Mistakes

**Mistake 1: Forgetting chmod**
```
-bash: ./script.sh: Permission denied
```
Fix: `chmod +x script.sh`

**Mistake 2: Wrong shebang**
```bash
#! /bin/bash    # space after # — this often fails
#!/usr/bin/bash # wrong path on some systems
```
Fix: Always use `#!/bin/bash` (no space, correct path confirmed with `which bash`)

**Mistake 3: Windows line endings (CRLF)**
```
/bin/bash^M: bad interpreter
```
Fix: In VS Code, click "CRLF" at bottom right → change to "LF" → save.

## Quick Recap

- Scripts are text files with commands
- Start every script with `#!/bin/bash`
- Use `chmod +x` to make it executable
- Run with `./scriptname.sh`
- Comments start with `#` — use them generously
