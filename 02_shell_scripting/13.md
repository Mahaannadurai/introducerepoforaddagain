
# Module 13 — Error Handling

## What You Will Learn
- What exit codes are and how to read them
- `set -e`, `set -u`, and `set -o pipefail`
- How to validate input before using it
- How to trap errors and clean up
- Writing scripts that fail safely

## Why This Matters

A script without error handling is like a car without brakes. Most of the time it's fine. But when something goes wrong — a missing file, a failed command, an empty variable — an unprotected script will silently continue, making the problem worse.

## Exit Codes

Every command in Linux produces an exit code when it finishes:
- `0` = success
- `1–255` = failure (different numbers mean different things)

```bash
ls /existing-folder
echo $?     # Prints: 0

ls /nonexistent-folder
echo $?     # Prints: 2 (no such file or directory)
```

Your scripts should also exit with meaningful codes:

```bash
exit 0    # Success
exit 1    # General error
exit 2    # Usage error (wrong arguments)
```

## Safe Script Settings

These three lines at the top of every serious script make it fail-safe:

```bash
#!/bin/bash
set -e           # Exit immediately if any command fails
set -u           # Treat unset variables as errors
set -o pipefail  # Catch failures inside pipes
```

Or combined:
```bash
set -euo pipefail
```

**Without these:**
```bash
#!/bin/bash
BACKUP_DIR="$UNDEFINED_VAR/backups"
rm -rf "$BACKUP_DIR"   # rm -rf /backups — deletes the wrong thing!
```

**With `set -u`:** The script stops at `$UNDEFINED_VAR` and prints an error instead of proceeding to the destructive command.

## Input Validation

```bash
#!/bin/bash

validate_inputs() {
    local TARGET_DIR="$1"

    if [ -z "$TARGET_DIR" ]; then
        echo "ERROR: No directory provided"
        echo "Usage: $0 <directory>"
        exit 2
    fi

    if [ ! -d "$TARGET_DIR" ]; then
        echo "ERROR: Directory does not exist: $TARGET_DIR"
        exit 1
    fi

    if [ ! -r "$TARGET_DIR" ]; then
        echo "ERROR: Cannot read directory: $TARGET_DIR"
        exit 1
    fi
}

validate_inputs "$1"
echo "Processing: $1"
```

## Guided Example — lib/validation.sh

```bash
#!/bin/bash

# lib/validation.sh
# Input validation functions for the InfraOps toolkit

require_argument() {
    local VALUE="$1"
    local NAME="$2"
    if [ -z "$VALUE" ]; then
        echo "ERROR: Required argument missing: $NAME" >&2
        exit 2
    fi
}

require_file() {
    local PATH_VAL="$1"
    if [ ! -f "$PATH_VAL" ]; then
        echo "ERROR: Required file not found: $PATH_VAL" >&2
        exit 1
    fi
}

require_directory() {
    local PATH_VAL="$1"
    if [ ! -d "$PATH_VAL" ]; then
        echo "ERROR: Required directory not found: $PATH_VAL" >&2
        exit 1
    fi
}

require_not_root() {
    if [ "$(id -u)" -eq 0 ]; then
        echo "ERROR: This script must not be run as root" >&2
        exit 1
    fi
}
```

## Trap — Clean Up After Errors

```bash
#!/bin/bash
set -euo pipefail

TEMP_FILE=$(mktemp)    # Create a temporary file

# Define cleanup function
cleanup() {
    rm -f "$TEMP_FILE"
    echo "Cleaned up temporary files"
}

# Trap: run cleanup on EXIT (whether success or failure)
trap cleanup EXIT

# Script work here...
echo "Working..." > "$TEMP_FILE"
cat "$TEMP_FILE"
# TEMP_FILE is automatically deleted when the script exits
```

`trap cleanup EXIT` means: "no matter how this script ends — normal exit, error, or Ctrl+C — run the cleanup function first."

## Quick Recap

- `$?` holds the last command's exit code
- `set -e` exits on any failure; `set -u` catches undefined variables; `set -o pipefail` catches pipe failures
- Always validate required inputs at the start of a script
- `trap function EXIT` runs cleanup on any exit
- Exit with meaningful codes: 0 = success, 1 = error, 2 = usage error

## Interview Corner — Error Handling

**Why interviewers ask:** Error handling separates professional scripts from hobby scripts. A script running in production needs to fail safely — not silently continue and make things worse.

**What they expect:**
- You know about `set -e`, `-u`, and `pipefail` and can explain each
- You validate arguments before using them
- You use `trap` for cleanup
- You exit with appropriate codes and messages

**Practice question:** "A script deletes files from a directory. How do you prevent it from accidentally deleting files from the wrong location?"

Expected answer includes: validate the variable isn't empty, validate the directory exists, confirm with the user if the directory is above a certain "danger level" in the filesystem, use `set -u` to prevent undefined variable expansion.
