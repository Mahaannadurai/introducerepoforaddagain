# Module 7 — Working With Multiple Choices

## What You Will Learn
- The `case` statement and when to use it over `if/elif`
- How to build menu-driven scripts
- How to match patterns in `case`

## Why This Matters

When a script has five or more possible conditions based on one variable, a chain of `elif` statements becomes hard to read. `case` is designed for exactly this situation — it's cleaner, faster to scan, and easier to extend.

## Real Linux Problem

InfraOps needs a single entry point: `toolkit.sh`. When you run it, a menu appears: 1) System info, 2) Disk check, 3) User audit, 4) Exit. Different choices run different scripts. This is a menu-driven toolkit — the kind every ops team uses.

## Simple Explanation

A `case` statement is like a router. A value comes in, and the router sends it to the right handler.

```
User types: "2"
      │
      ▼
case "2" in
   "1") run system-info ;;
   "2") run disk-check   ;;   ← matches here
   "3") run user-audit   ;;
   *)   show error        ;;
esac
```

`esac` is `case` backwards — closes the case block (just like `fi` for `if`).

## case Syntax

```bash
case "$VARIABLE" in
    "option1")
        # commands
        ;;
    "option2")
        # commands
        ;;
    "option3"|"option4")  # Multiple options for same block
        # commands
        ;;
    *)
        # Default: runs if nothing else matched
        ;;
esac
```

`;;` ends each option block — think of it as the "break" that stops the case from falling through.

## Guided Example — toolkit.sh

This becomes the main entry point for the whole `devops-bash-toolkit` project.

```bash
#!/bin/bash

# toolkit.sh
# InfraOps Bash Toolkit — main menu
# Runs from the project root

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"

show_menu() {
    echo ""
    echo "============================================"
    echo "  InfraOps Bash Toolkit"
    echo "============================================"
    echo "  1) System information"
    echo "  2) Disk usage check"
    echo "  3) Server snapshot"
    echo "  4) Environment check"
    echo "  0) Exit"
    echo "============================================"
    echo ""
    read -p "Select an option: " CHOICE
}

show_menu

case "$CHOICE" in
    1)
        echo "Running system information..."
        bash "$SCRIPT_DIR/scripts/system/system-info.sh"
        ;;
    2)
        echo "Running disk check..."
        bash "$SCRIPT_DIR/scripts/system/disk-check.sh"
        ;;
    3)
        echo "Taking server snapshot..."
        bash "$SCRIPT_DIR/scripts/system/server-snapshot.sh"
        ;;
    4)
        echo "Running environment check..."
        bash "$SCRIPT_DIR/scripts/system/env-check.sh"
        ;;
    0)
        echo "Goodbye!"
        exit 0
        ;;
    *)
        echo "Invalid option: $CHOICE"
        echo "Please run again and choose 0-4."
        ;;
esac
```

```bash
chmod +x ~/devops-bash-toolkit/toolkit.sh
cd ~/devops-bash-toolkit
./toolkit.sh
```

## Common Mistakes

**Forgetting `;;` at the end of each option:**
```bash
case "$X" in
    "a")
        echo "A"    # Missing ;; — falls through to next option!
    "b")
        echo "B"
        ;;
esac
```

**Forgetting `*)`:**
Without the default case, unrecognized input is silently ignored. Always include `*)` to handle unexpected values.

## Quick Recap

- `case "$VAR" in ... esac` matches a variable against multiple options
- Each option ends with `;;`
- `option1|option2)` matches either value
- `*)` is the catch-all default
